# Architecture map

Full end-to-end reference for `on-device-browser-agent`, built from a complete read-through of `extension/src/` on 2026-07-02 (right after the `prod-readiness-review-fixes` merge), updated 2026-07-03 to cover the `agent/framework/` seat layer and model-tiering work merged since, and again 2026-07-05 to cover tool-execution reliability, anti-bot detect-and-pause, and the side-panel progress/status work (all three independent of, and unrelated to, the framework/tiering work). `CLAUDE.md` has the short version and the highest-value gotchas; this is the detailed trace. If something below looks stale, trust the code over this doc and update it.

## Component map

- **`sidepanel/`** (React) — the only UI. One `chrome.runtime.Port` (`sidepanel/port.ts`) to the background; renders state, forwards commands, treats agent logic as opaque.
- **`background/`** — the MV3 service worker. Owns the single `Orchestrator` instance and the Ollama HTTP client, persists run/timeline state, brokers every message from the panel.
- **`agent/`** — `orchestrator.ts` (the loop) + `roles/{planner,executor,evaluator,compactor}.ts` + `prompts/index.ts` (the literal prompt text, including the prompt-injection defense).
- **`agent/framework/`** — the "kitchen" seat layer between the orchestrator and `roles/*.ts`: `head_chef.ts`, `sous_chef.ts`, `helper.ts` are thin named wrappers around `runPlanner`/`runEvaluator`/`runExecutor`+`runCompactor` respectively (no logic moved except grounding — see below); `provider.ts` holds the `ModelProvider` interface and every model-backend concern (local Ollama, Anthropic, any OpenAI-compatible endpoint, fallback, thinking override, resolution). See "Model tiering" below.
- **`agent/tools/`** — the tool registry + CDP-driven browser actions (`tools/browser/*`) + the ARIA-tree perception pipeline.
- **`agent/safety/`** — domain tiers, PII redaction, circuit breaker. Three independent, composable checks, not one gate.
- **`agent/facts.ts`, `agent/verify/grounding.ts`, `agent/workflow_memory.ts`, `agent/budget.ts`** — cross-cutting: the grounded facts ledger, the anti-hallucination verifier, learned/seeded multi-step "recipes," and per-role context-window budgeting.

## A goal's journey, end to end

1. User types a goal → side panel sends `{type:'agent.start', goal}` over the port.
2. `background/index.ts`'s `handleStart`: preflights Ollama (`ping` + `listModels`), builds a tool registry + fresh `Orchestrator`, starts a **20-second keepalive** (a cheap `chrome.runtime.getPlatformInfo()` tick — Chrome only resets the SW idle timer on an extension-API touch, not on an open port or a pending fetch), then calls `orch.start(goal)` then `runUntilTerminal()` **without awaiting**. This "detached run loop" exists because MV3 force-kills a single `onMessage` handler after 5 minutes, but a multi-step task can run far longer — returning immediately ends that "event" so the run continues as an unbounded task, kept alive only by the keepalive ping. A monotonic `_runId` guards every mutation of shared state (`_orch`, `_events`, keepalive) against a stale, already-superseded run stomping on a newer one.
3. `Orchestrator.start()` resets all per-task state and matches the goal against stored workflow recipes (`matchWorkflow` — keyword-overlap scoring gated by a discriminating `requiredAny` token so an unrelated goal can't hijack a recipe).
4. **Plan** — the head-chef seat (`runHeadChef`, wrapping `runPlanner` unchanged) is called once. If a recipe matched, its steps are injected as a "PROVEN RECIPE" the planner is told to build from verbatim; a bounded (once-per-task) retry catches the model collapsing a multi-step recipe into fewer steps than it has.
5. **The loop is not round-robin.** Every turn calls the helper seat (`runHelper`, wrapping `runExecutor`) for exactly one tool call. The sous-chef seat (`runSousChef`, wrapping `runEvaluator`) only runs when a step advances/finishes, or periodically (every 3rd turn) as a drift safety net. The head-chef only re-runs via `replan()` when the circuit breaker trips or the evaluator sets `shouldReplan`, bounded by `maxReplans` (default 3). The compactor (`runHelperCompaction`, wrapping `runCompactor`) only runs when the executor's prompt would cross ~80% of its token budget *and* the scratchpad exceeds 1,000 chars.
6. **Tool dispatch**: `ToolRegistry.dispatch` validates args against a Zod schema, then the tool's own `dispatch()` runs. Every *mutating* browser tool (`tab.click`/`tab.type`/`tab.select`/`tab.upload_file`) calls `assertCanAct(url, 'click-only', ...)` from the safety layer **as the first line inside the tool itself** — this is defense in depth (no central enforcement point), which also means a new mutating tool that forgets the call would silently bypass the system. Browser actions run through `withCdp`: attach `chrome.debugger` (one retry, 10s timeout), send commands (20s timeout each — an unbounded command once froze the agent for 9 minutes per the code comment), always detach in a `finally`.
7. **After any navigating action** (click/open/a submitting type), the orchestrator auto-re-reads the page for the model and opportunistically auto-dismisses a consent/cookie wall if the domain is already tier-approved — compensation for a small model reliably forgetting to re-observe after it changes the page. An **observe-then-act gate** blocks calling the same observation tool twice in a row within one step.
8. **Finishing** always routes through grounding: `verifyFinish` (now living in `agent/framework/sous_chef.ts` — "does this hold up" is the sous chef's job, moved out of the orchestrator; logic is unchanged) extracts every number-like token from the proposed answer and checks set-membership against a **grounding corpus** (everything read this run + the facts ledger). A claim with an ungrounded number is downgraded to `partial` with an `[unverified against page]` tag, never silently asserted. A "field is missing" claim is separately double-checked against the full corpus before being trusted (`reconcileMissingFromCorpus`). Give-up and denial paths try to **salvage** an answer from everything already read (`giveUp`/`preferSalvageOnDenial` → `synthesizeFromObserved`) before reporting empty failure — but only when a real action denial occurred; an honest "not found" with no denial is never overridden. The mid-plan prose-answer gate (`orchestrator.ts`, around the point a non-final step's prose answer is accepted) also routes through `verifyFinish` now, as of a 2026-07-03 follow-up — grounding lives entirely in the sous-chef seat, no duplicate call site left calling `groundingCorpus`/`ungroundedNumbers` directly.
9. **Recipe learning**: a clean run (no replan, no evaluator FAIL, no tier/fatal denial, no breaker trip, no finish rejection — tracked by a sticky `runDirty` flag) with real page interaction and no redundant repeats can be auto-learned as a new recipe. A recipe-driven run that wasn't clean gets quarantined: `user` recipes roll back to a `lastGood` snapshot (or are deleted if none exists yet); `auto` recipes are deleted outright — one proven run before they're disposable; `builtin` recipes are immutable.
10. **Every emitted/persisted event** passes through one redaction chokepoint (`Orchestrator.emit` → `redactEvent`) before being broadcast to the panel and written to IndexedDB.

## Model tiering — the seat/provider layer (`agent/framework/`)

The orchestrator doesn't call `runPlanner`/`runExecutor`/`runEvaluator`/`runCompactor` directly anymore — it calls `runHeadChef`/`runHelper`/`runSousChef`/`runHelperCompaction` (`agent/framework/{head_chef,sous_chef,helper}.ts`), each a thin named wrapper that forwards to the same underlying `roles/*.ts` function unchanged, just swapping in a `provider: ModelProvider` (`{chatOnce(opts): Promise<ChatResponse>}`) instead of a hardcoded `OllamaClient`. `roles/*.ts` kept their names, exports, and all prompt/parsing/retry logic — nothing moved except grounding (see above).

Two constructor fields drive it (`orchestrator.ts`): `leadProvider` (used by `runHeadChef`/`runSousChef` — planner and evaluator) and `helperProvider` (used by `runHelper`/`runHelperCompaction` — executor and compactor, **always** `localProvider(ollama)`, never frontier-eligible). `leadProvider = resolveLeadProvider(settings, ollama, onFallback)`:

- `settings.hybridMode` off, or no `settings.frontier.apiKey` → `localProvider(ollama)` (an identity function — `OllamaClient` already structurally satisfies `ModelProvider`). This is the byte-identical-to-local-only default, not a fallback layered on top of it.
- Otherwise → `frontierProviderFor(settings.frontier)` (dispatches on `settings.frontier.provider`) wrapped in `withFallback(..., localProvider(ollama), onFallback)`: one retry on a 5xx, then falls back to local and fires `onFallback` — wired to a visible `{kind:'log', level:'warn'}` emit, never a silent swap.
- Either way, the result is wrapped once more in `withThinkingOverride(base, settings.leadThinking)` — `undefined` is a true no-op (returns the same object reference); `true`/`false` forces `ChatOptions.thinking` on every call through whichever provider ended up underneath, local or frontier.

`settings.frontier` (`FrontierConfig`, `shared/messages.ts`) is a discriminated union: `{provider:'anthropic', apiKey, model}` hits Claude's Messages API directly (`frontierProvider` — raw `fetch`, no SDK, `x-api-key` header, splits a `system` message out to a top-level field, `thinking:{type:'adaptive'|'disabled'}`); `{provider:'openai-compatible', apiKey, model, baseUrl}` hits any user-supplied URL speaking the OpenAI Chat Completions shape (`openAICompatibleProvider` — `Authorization: Bearer` header, `system` role stays inline in `messages`, `reasoning_effort:'high'` sent only when thinking is explicitly on) — one implementation covers OpenAI itself, OpenRouter, DeepSeek, MiniMax, and self-hosted servers, distinguished only by config, never new code. Both throw a status-bearing error on a non-2xx response, on a model refusal, and (as of a 2026-07-03 follow-up) on an empty/missing text field — no silent `rawText: ''` on a malformed body. The API key is never interpolated into a prompt, an error message, or an `emit()` payload — it only ever appears in a request header, so there's no channel for it to leak through the existing PII-redaction chokepoint (which wouldn't recognize it as PII anyway).

The Settings UI (`sidepanel/components/SettingsPanel.tsx`, "Frontier model (optional)" card) exposes: a `hybridMode` checkbox with a cost disclaimer ("calls a paid API repeatedly... doesn't track or cap that spend"); a Provider select (Anthropic / OpenAI-compatible); Model + API key fields with provider-appropriate placeholders; a Base URL field (openai-compatible only, prefilled with OpenAI's own endpoint, editable, with OpenRouter/DeepSeek examples in the hint text); and a Thinking select (Default/Always on/Always off) that sits *outside* the `hybridMode`-conditional block since it's meaningful in local-only mode too. `updateFrontier` remembers a typed custom `baseUrl` in its own piece of local state, decoupled from whichever provider is currently selected, specifically so it survives a round-trip through the Provider dropdown (a real bug, found and fixed during implementation — see the 2026-07-03 plan doc's Task 2 for the exact failure mode).

## Tool layer & ARIA perception

Full catalog (`agent/tools/index.ts`): `echo`, `next_step`, `finish`, `memory.read`/`memory.write`/`memory.list` (blocks the reserved `awm:` key prefix used internally for recipe storage — this check is hand-reimplemented in three tool files, not a shared helper), `tab.open`, `open_result` (opens a numbered search result without the model retyping a URL — added because the model kept fabricating plausible-looking 404 paths), `tab.close`, `tab.list`, `tab.wait_loaded`, `tab.screenshot`, `aria.extract`, `tab.read_active`, `search`, `vision.read` (screenshot + multimodal fallback when the a11y tree is too thin), `tab.click`, `tab.type`, `tab.select`, `tab.scroll`, `tab.upload_file`.

- **ARIA simplification** (`tools/browser/aria.ts`): recursively drops layout-only nodes, splices unnamed wrapper nodes up a level, collapses single-child wrapper chains, assigns a sequential integer index to every interactive node, and caps serialized text at 12,000 chars via escalating trims that specifically protect form inputs even inside otherwise-collapsed nav/footer landmarks. The index→node cache is keyed by tab+URL; invalidated on navigation and unconditionally after every successful mutating action — "act, then re-read" is a structural assumption, not a live DOM diff.
- **`tab.scroll`** has no domain-tier gate (it's read-only) but clears the extraction cache unconditionally after every scroll and reports honest before/after `scrollY` deltas — infinite-scroll/lazy-load pages can mutate the DOM, so the cache is invalidated defensively rather than diffed. This is deliberate, not a bug (a stale code comment once asserted scrolling "changes nothing"; a 2026-07-01 review corrected the comment but left the behavior as-is since no live failure had reproduced a problem).
- **Upload** (`tools/browser/upload.ts`): Chrome blocks `DOM.setFileInputFiles` for extensions, so stored résumé bytes are rebuilt into a `File` client-side and injected via a `DataTransfer` assigned to `input.files`, dispatching `input`+`change` with `composed:true`. The input-finder walks light DOM and **open** shadow roots (closed roots are unreachable to any DOM API, not a bug); cross-origin iframes aren't traversed at all. Never submits the form.
- **Search** (`tools/browser/search.ts`): no API key. Scrapes DuckDuckGo's html/lite endpoints, detects being blocked (captcha/anomaly page) distinctly from a genuine empty result, and as a last resort opens an invisible backgrounded Google tab (full cookie/JS session, so it isn't served a captcha) and scrapes via a per-engine selector cascade.
- Every action tool checks `isElementConnected` before acting and returns an explicit "stale, re-extract" message rather than silently no-op'ing. `tab.select` reads its value back after assignment since an invalid option is a silent no-op otherwise. Clicks use native `.click()`, not synthetic mouse events, because synthetic coordinates didn't reliably navigate on background tabs.
- **Action reliability, added 2026-07-04** (`tools/browser/actions.ts`): `tab.click` on a checkbox/radio/switch reads the toggled state back after clicking and retries once via the element's associated `<label>` if the direct click didn't register (reported in the timeline as a retry, not silently); the coordinate-fallback click path (only reached when `DOM.resolveNode` itself fails, so there's no in-page object reference to compare against) checks CDP's `DOM.getNodeForLocation` for occlusion first, refusing with an actionable message rather than clicking whatever's actually on top — and only refuses on positive evidence of occlusion, falling back to the unconditional dispatch if the check itself throws. `tab.type`'s `clear` step and its new date/time/color/range branch both go through the native property-descriptor setter (bypasses a React-controlled field's own value-tracking interception) instead of a raw `this.value =` assignment, with the plain-assignment path kept as a fallback if the descriptor lookup itself throws. A plain-text `tab.type` additionally reads the value back after typing and retries once (native-setter clear, then retype) specifically when the read-back looks like leftover-text concatenation, not for any other kind of mismatch (e.g. an input mask reformatting the value is left alone). `tab.select` gained a second path for ARIA `combobox`+`listbox` custom dropdowns (expand → match the target listbox's options by trimmed text, case-insensitively → click → verify the displayed value changed → collapse via Escape) alongside the original native-`<select>` path, which is unchanged. See `docs/superpowers/specs/2026-07-04-tool-execution-reliability-design.md` for the full design, including the type-lookup decision (a narrow, zero-extra-round-trip JS property read at act time, not a bulk `DOMSnapshot` merge at perception time) and what's deliberately still out of scope (password-value redaction, other custom-dropdown shapes, date-format hints).

## Safety layer

- **Domain tiers** (`agent/safety/domain_tiers.ts`): two tiers, `read-only` (default, ordinal 0) and `click-only` (ordinal 1). Unknown/unparseable hosts fail closed to `read-only`. `assertCanAct(url, required, tiers, bypass)` checks in this exact order: (1) `isBlockedUrl` — throws unconditionally, before bypass is even read; (2) `if (bypass) return` — skips the tier comparison only; (3) tier comparison. `BLOCKED_PROTOCOLS` = `file:, chrome:, chrome-extension:, devtools:, view-source:, javascript:, data:, blob:, about:, ws:, wss:` — no setting can relax this set. A `full-action` third tier existed pre-2026-07-01, was confirmed byte-identical to `click-only` in enforcement, and was removed; `migrateLegacyTier` reads old persisted `full-action` values back as `click-only` so existing installs aren't silently narrowed. **The same blocklist-then-bypass ordering is composed at a second call site**, `orchestrator.ts`'s private `canActUrl` (gates the consent-auto-dismiss feature only) — it calls the same shared `isBlockedUrl`/`getDomainTier`/`hostFor`/`TIER_ORDER` primitives, so the underlying check can't itself diverge, but the *ordering* is written twice; a future change to one wouldn't automatically apply to the other.
- **Redaction** (`agent/safety/redact.ts`): one shared recursive walker (`redactAny`) backs both `redactEvent` (values only, depth-capped at 8, used on every `TimelineEvent`) and `redactDeep` (keys and values too, uncapped, used on tool findings and the facts ledger). Patterns: Luhn-validated credit card, SSN, email, phone, address — regex/heuristic, so it won't catch PII outside those five shapes (names, DOB, IBANs, etc. are an inherent, undocumented gap). The recursive-walker design specifically fixed a historical bug class: numbers are never stringified before hitting the regex engine, so a `Date.now()` timestamp can no longer false-positive as a credit-card candidate (it used to, ~10% of the time, under the old per-field checklist approach). Redaction happens only at the persistence/telemetry boundary — user profile data is deliberately sent to the LLM **unredacted** for job-apply autofill to work; it's only scrubbed going into IndexedDB/UI events.
- **Circuit breaker** (`agent/safety/circuit_breaker.ts`): pure, stateless-transition functions. Four independent trip conditions checked every turn: 3 consecutive identical actions (exact tool+args hash match — any argument variation evades this), fewer than 3 distinct actions in the last 10 (low diversity), 3+ unknown-tool calls in 8 turns (storm), or 8 turns with no new finding (no progress). `resetForNewStep` discards all windowed counters at step boundaries so a near-miss on one step can't wrongly cut short the next.
- **Prompt-injection defense**: every role's system prompt marks `<untrusted_page_content>` as data, not instructions, applied at one chokepoint (`wrapPageContent` in `prompts/index.ts`).

## Anti-hallucination system

The most deliberate design story in the codebase. `verifyFinish` (`agent/framework/sous_chef.ts` — relocated verbatim from `orchestrator.ts` during the framework refactor, logic unchanged) is **deterministic number-matching**, not another LLM call: normalizes numeric tokens (bare single digits like list markers are exempt) and checks set-membership against a **grounding corpus** = observed-text FIFO (~60KB rolling window) + the facts ledger. An LLM-based "does this hold up against the page" verifier was built and benchmarked, then dropped, because it false-rejected correct/honest answers and dropped the benchmark's `correct` rate from 80% to 67% (`docs/superpowers/specs/2026-06-18-theme-a-page-grounded-verification-design.md`). This is deliberately dumb in a specific way: it's true set-membership on normalized numbers, not substring search, so "4.6" is never wrongly grounded by a page saying "14.62" — but non-numeric hallucinations (wrong names, wrong facts with no digits) are entirely unguarded by this mechanism.

The **facts ledger** (`agent/facts.ts`) holds ≤24 entries, each admitted only if every number inside it is independently grounded at admission time, deduped by exact text, FIFO-evicted. It exists specifically so a fact survives even after its source page scrolls out of the 60KB observed-text window. `mentionsMissing`'s regex (detecting "this field isn't listed" claims) has an unusually long comment trail documenting three rounds of tightening to avoid false-suppression, with one residual false-positive gap left deliberately unfixed (documented, not an oversight).

## Workflow-memory recipes

`agent/workflow_memory.ts` — three origins: `builtin` (`SEED_WORKFLOWS`, 18 seed recipes, curated, immutable), `user` (authored via the Recipes UI tab, gets a `lastGood` rollback snapshot), `auto` (learned from a trace, one proven run before disposable, no rollback). `matchWorkflow` scores by keyword/domain overlap with a discriminator gate against false matches. A run is only eligible to *become* a recipe if it was clean (see step 9 above) and its trace isn't redundant (no repeated identical search/open) — a complete user-facing success can still fail to produce or reinforce a recipe if it needed any corrective machinery along the way. Learned-recipe goal text is redacted for PII before persisting (job-apply goals routinely carry email/phone/name).

## Budget & context window

`agent/budget.ts`: `DEFAULT_NUM_CTX` 32,768, range 8,192–131,072, user-configurable in Settings. Per-role token budgets scale with the window, but the **per-page read cap is fixed at 12,000 chars regardless of `num_ctx`** — the stated rationale is that a bigger window should let the agent remember more across turns, not read one page more sprawlingly ("lost in the middle"). Scratch/observed/salvage caps do scale with the window. No runtime guard exists against setting `num_ctx` too high for the user's actual VRAM — that's a documentation-only warning (check `ollama ps` after raising it); misconfiguring it can break every task with no in-app validation. `metrics.ts` only times the 4 LLM role calls, not individual CDP/tool actions, which limits its use for diagnosing whether slowness is the model or the browser layer.

## Job-apply flow

Résumé upload → text extraction (`sidepanel/resume.ts`: mammoth for `.docx`, pdfjs for `.pdf`, 20MB cap; scanned/image PDFs yield empty text with no OCR fallback, surfaced to the user as an explicit "needs OCR" message, no remediation path) → one-shot LLM call extracts profile JSON (`agent/profile.ts`, validation is hand-rolled type/array/empty-value filtering — no Zod, despite Zod being used elsewhere) → user reviews and must explicitly click Save (not auto-persisted). Paste a Greenhouse/Lever URL → `sidepanel/apply.ts` builds a goal string engineered to keyword-match the `seed-job-application` recipe and to textually forbid submission → the agent fills the form and attaches the résumé (same `DataTransfer` injection as general upload) → **stops before submit**.

**That stop is enforced only by prompt/recipe wording — not structurally.** No `submit` tool exists, but generic `tab.click`/`tab.type(..., submit: true)` do, and nothing prevents clicking a button labeled Submit; `ToolContext` doesn't carry a goal or task-type flag, so no tool can check "this is a job-apply run" before acting. The guarantee is entirely prompt discipline plus the generic domain-tier gate. Not yet supported at all: Workday's drag-drop/direct-to-S3 uploader, and forms embedded in a cross-origin iframe.

## Anti-bot detect-and-pause (added 2026-07-05)

`detectAntiBotBlock` (`agent/tools/browser/antibot.ts`) — deterministic, ordered regex matching against the observed page text, modeled directly on the existing `findConsentDismiss`: vendor-specific patterns (Cloudflare's "checking your browser," Google's "unusual traffic," Akamai's "pardon the interruption," CAPTCHA widget names) checked before generic human-verification phrasing, so a positive vendor ID takes precedence. A miss is a false negative (no pause, never a wrong action); a false positive is fully recoverable via Stop, never a silent hang — the design explicitly favors under- over over-triggering.

Checked from `orchestrator.ts`'s post-navigation hook, right where the existing consent-dismiss check already runs. On a match: emits `{kind:'antibot.blocked', label}`, sets `TaskPhase` to a new `'BLOCKED'` value, then loops — sleep `antiBotPollMs` (default 5,000ms, an `OrchestratorOpts` field), re-extract, re-check — with **no timeout**, until the pattern stops matching. On resolution: emits `antibot.resolved`, records the now-clear page as observed, and returns to `'EXECUTING'` automatically, no user action needed to resume. `isRunning()` (`sidepanel/view/phase.ts`) treats `BLOCKED` as a running/busy phase, so the side panel's Stop button is the only manual escape hatch — there is no auto-dismiss, click-through, or solve path anywhere in this feature, by design.

**This is a detect-and-pause feature only, deliberately and permanently — not a step toward solving anything.** The user twice asked for the agent to solve/bypass anti-bot challenges (CAPTCHAs, sliders, puzzles, broadened on request to any anti-bot interstitial); both requests were declined on the same reasoning: defeating bot detection is categorically different from making a legitimate action more reliable, regardless of stated intent, and even feature-comparable commercial tools don't ship this locally. This pause-and-wait behavior is the accepted alternative the user asked for instead. See `docs/superpowers/specs/2026-07-05-antibot-pause-design.md`'s Non-goals section for the full reasoning trail — don't reopen the solve/bypass question without re-reading it first.

## Side panel UI

Single `chrome.runtime.Port` (`sidepanel/port.ts`): `send()` lazily calls `chrome.runtime.connect()` if the port is dead — this both revives a dropped connection and wakes an idle-killed service worker, since `connect()` itself wakes MV3 SWs. Reconnection is on-demand only, never polled — the one exception is a 3.5s Ollama-reachability poll, justified because the extension can't launch `ollama serve` itself. `App.tsx` carries an explicit "contract: do not change message shapes" comment over the connection block. Timeline events are capped at 1,000 to bound memory on long runs. Steering (`agent.steer`) is fire-and-forget from the UI's perspective — no `SwUpdate` confirms it was received.

**Live-progress signal, added 2026-07-05.** `RunState` derives a step-completion progress meter from `planProgress()` (`sidepanel/view/result.ts` — `{total, done, activeIndex}` from the current `Plan`), a filled bar paired with a live activity-event count so a short plan sitting on one long-running step still shows real, moving evidence of work rather than a frozen 0%-until-the-end bar. The same status (phase + step fraction + action count) is duplicated into the composer bar itself (`Composer.tsx`'s `composer-status`), since the composer is the one part of the page guaranteed visible regardless of scroll position — `RunState` sits above the Activity log in the DOM, and the panel's chat-style auto-scroll-to-bottom (`App.tsx`) can otherwise push it out of view once the log grows past one viewport height.

**A real bug found and fixed in the same pass:** `pushStatus()` was only ever broadcast twice per task — once right after `orch.start()` (phase still `'IDLE'`, per `state_store.ts`'s `_setHot`), and once more in `handleStart`'s `finally` block, *after* `runUntilTerminal()` had already resolved to `'DONE'`/`'ABORTED'`. Every real phase transition in between (`PLANNING`→`EXECUTING`→`EVALUATING`→...) happens via `patchHot()` deep inside the run loop with no broadcast of its own, so `isRunning()` was structurally always `false` from the panel's perspective — Stop never showed in place of Run, for any task, regardless of duration, and neither the progress meter nor the composer status line (both correct in isolation) ever had a `running===true` render to appear in. Fixed by pushing status from `appendEventLocal` — the same per-event path the Activity log's `append` broadcast already uses — so status now stays exactly as fresh as the timeline, with no new plumbing. `tests/unit/background_run_lifecycle.test.ts`'s `bg.addTestPanel()`/`fakeOrch()` harness (captures the real `emit` callback passed to the `Orchestrator` constructor, so a test can call it directly to simulate a mid-run event) is the right tool for testing broadcast timing itself, not just downstream data effects.

## Testing, build, bench

- **Unit/integration/property** (`npm test`): vitest + happy-dom + `fake-indexeddb/auto`, `chrome.*` stubbed in-memory in `tests/setup.ts` — no real Chrome needed. Coverage is heaviest on background plumbing (~11 files) and tools/ARIA/upload (~15 files); thinnest on the side panel UI (2 files, one of which — `components_render.test.tsx` — now includes a genuinely interactive `createRoot`/`act`-based test for the Settings UI's provider-switch state handling, not just static-markup rendering). The `agent/framework/` layer has its own dedicated unit test per file (`tests/unit/framework_{provider,head_chef,sous_chef,helper}.test.ts`), plus an `orchestrator.test.ts` integration test proving hybrid-mode seat routing never swaps the lead/helper providers.
- **Bench** (`npm run bench`, opt-in via `OLLAMA_BENCH=1`): `ScriptedBrowser` fakes CDP via a per-task transition-state-machine (no real Chrome), replays typed field values back into the ARIA string so evaluators can verify a form was actually filled. Runs the **real** `Orchestrator` + **real** local Ollama model against scripted fixtures; `scorer.ts` computes `completed`/`correct`/`grounded` (grounded = passes `ungroundedNumbers` *and* has no leaked-but-unobserved entities/forbidden strings — number-grounding alone can't catch single-digit fabrications like an invented "5 stars").
- `scripts/measure_toolcalls.mjs` and the bench are two different things: the former measures model-only tool-call syntax reliability (no orchestrator); the latter measures full-loop task correctness.
- `scripts/browser_smoke.py` is a separate Selenium sanity sweep against a real, built Chrome extension — light assertions, not a full harness.

## Where design rationale lives

## Recurring design principles

1. **Mechanical beats semantic for this model size.** Every place a ~4B model was asked to make a judgment call it's bad at, the fix replaced the judgment with a rule requiring none: number-grounding instead of LLM-verify; same-domain-anchoring instead of asking the model to classify a comparison's basis; a hardcoded protocol blocklist instead of a prompt rule.
2. **Evidence over a-priori design.** The LLM finish-verifier wasn't dropped speculatively — it was built, benchmarked, measured to regress `correct` 80%→67%, then dropped. Changes to grounding/safety logic in this repo are expected to clear `npm run bench`, not just "sound right."
3. **Fail closed, degrade loud.** Unknown hosts → read-only. Unparseable URLs → blocked. Ungrounded numbers → downgraded with the reason appended inline, never silently asserted. A stale DOM node → an explicit re-read message, never a silent no-op. An anti-bot block → pause and wait for a human, never auto-dismiss or solve.
4. **Defense in depth, with a real duplication cost.** Domain-tier checks live inside individual tool bodies, not only behind a central gate — so a bug in one layer doesn't remove the other, at the cost of a forgetful future tool silently bypassing the system, and the blocklist-ordering being written at two call sites (see Safety layer above).
5. **Trust calibrated to investment.** Auto-learned recipes (cheap to produce) are deleted on first friction. User-authored recipes (a human's effort) roll back to last-known-good instead. Built-in recipes are untouchable.
6. **Optional features prove "unchanged when off" structurally, not by claim.** `resolveLeadProvider` returns the exact same `localProvider(ollama)` reference when hybrid mode is off; `withThinkingOverride` returns the exact same reference when `leadThinking` is unset. Both are backed by identity-equality tests (`toBe`, not `toEqual`) and an explicit end-to-end regression test reusing the pre-feature baseline's fixtures — the guarantee falls out of the code, it isn't a comment asserting it does.
