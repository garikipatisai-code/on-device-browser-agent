# Architecture map

Full end-to-end reference for `on-device-browser-agent`, built from a complete read-through of `extension/src/` on 2026-07-02 (right after the `prod-readiness-review-fixes` merge). `CLAUDE.md` has the short version and the highest-value gotchas; this is the detailed trace. If something below looks stale, trust the code over this doc and update it.

## Component map

- **`sidepanel/`** (React) — the only UI. One `chrome.runtime.Port` (`sidepanel/port.ts`) to the background; renders state, forwards commands, treats agent logic as opaque.
- **`background/`** — the MV3 service worker. Owns the single `Orchestrator` instance and the Ollama HTTP client, persists run/timeline state, brokers every message from the panel.
- **`agent/`** — `orchestrator.ts` (the loop) + `roles/{planner,executor,evaluator,compactor}.ts` + `prompts/index.ts` (the literal prompt text, including the prompt-injection defense).
- **`agent/tools/`** — the tool registry + CDP-driven browser actions (`tools/browser/*`) + the ARIA-tree perception pipeline.
- **`agent/safety/`** — domain tiers, PII redaction, circuit breaker. Three independent, composable checks, not one gate.
- **`agent/facts.ts`, `agent/verify/grounding.ts`, `agent/workflow_memory.ts`, `agent/budget.ts`** — cross-cutting: the grounded facts ledger, the anti-hallucination verifier, learned/seeded multi-step "recipes," and per-role context-window budgeting.

## A goal's journey, end to end

1. User types a goal → side panel sends `{type:'agent.start', goal}` over the port.
2. `background/index.ts`'s `handleStart`: preflights Ollama (`ping` + `listModels`), builds a tool registry + fresh `Orchestrator`, starts a **20-second keepalive** (a cheap `chrome.runtime.getPlatformInfo()` tick — Chrome only resets the SW idle timer on an extension-API touch, not on an open port or a pending fetch), then calls `orch.start(goal)` then `runUntilTerminal()` **without awaiting**. This "detached run loop" exists because MV3 force-kills a single `onMessage` handler after 5 minutes, but a multi-step task can run far longer — returning immediately ends that "event" so the run continues as an unbounded task, kept alive only by the keepalive ping. A monotonic `_runId` guards every mutation of shared state (`_orch`, `_events`, keepalive) against a stale, already-superseded run stomping on a newer one.
3. `Orchestrator.start()` resets all per-task state and matches the goal against stored workflow recipes (`matchWorkflow` — keyword-overlap scoring gated by a discriminating `requiredAny` token so an unrelated goal can't hijack a recipe).
4. **Plan** — the Planner is called once. If a recipe matched, its steps are injected as a "PROVEN RECIPE" the planner is told to build from verbatim; a bounded (once-per-task) retry catches the model collapsing a multi-step recipe into fewer steps than it has.
5. **The loop is not round-robin.** Every turn calls the Executor for exactly one tool call. The Evaluator only runs when a step advances/finishes, or periodically (every 3rd turn) as a drift safety net. The Planner only re-runs via `replan()` when the circuit breaker trips or the evaluator sets `shouldReplan`, bounded by `maxReplans` (default 3). The Compactor only runs when the executor's prompt would cross ~80% of its token budget *and* the scratchpad exceeds 1,000 chars.
6. **Tool dispatch**: `ToolRegistry.dispatch` validates args against a Zod schema, then the tool's own `dispatch()` runs. Every *mutating* browser tool (`tab.click`/`tab.type`/`tab.select`/`tab.upload_file`) calls `assertCanAct(url, 'click-only', ...)` from the safety layer **as the first line inside the tool itself** — this is defense in depth (no central enforcement point), which also means a new mutating tool that forgets the call would silently bypass the system. Browser actions run through `withCdp`: attach `chrome.debugger` (one retry, 10s timeout), send commands (20s timeout each — an unbounded command once froze the agent for 9 minutes per the code comment), always detach in a `finally`.
7. **After any navigating action** (click/open/a submitting type), the orchestrator auto-re-reads the page for the model and opportunistically auto-dismisses a consent/cookie wall if the domain is already tier-approved — compensation for a small model reliably forgetting to re-observe after it changes the page. An **observe-then-act gate** blocks calling the same observation tool twice in a row within one step.
8. **Finishing** always routes through grounding: `verifyFinish` extracts every number-like token from the proposed answer and checks set-membership against a **grounding corpus** (everything read this run + the facts ledger). A claim with an ungrounded number is downgraded to `partial` with an `[unverified against page]` tag, never silently asserted. A "field is missing" claim is separately double-checked against the full corpus before being trusted (`reconcileMissingFromCorpus`). Give-up and denial paths try to **salvage** an answer from everything already read (`giveUp`/`preferSalvageOnDenial` → `synthesizeFromObserved`) before reporting empty failure — but only when a real action denial occurred; an honest "not found" with no denial is never overridden.
9. **Recipe learning**: a clean run (no replan, no evaluator FAIL, no tier/fatal denial, no breaker trip, no finish rejection — tracked by a sticky `runDirty` flag) with real page interaction and no redundant repeats can be auto-learned as a new recipe. A recipe-driven run that wasn't clean gets quarantined: `user` recipes roll back to a `lastGood` snapshot (or are deleted if none exists yet); `auto` recipes are deleted outright — one proven run before they're disposable; `builtin` recipes are immutable.
10. **Every emitted/persisted event** passes through one redaction chokepoint (`Orchestrator.emit` → `redactEvent`) before being broadcast to the panel and written to IndexedDB.

## Tool layer & ARIA perception

Full catalog (`agent/tools/index.ts`): `echo`, `next_step`, `finish`, `memory.read`/`memory.write`/`memory.list` (blocks the reserved `awm:` key prefix used internally for recipe storage — this check is hand-reimplemented in three tool files, not a shared helper), `tab.open`, `open_result` (opens a numbered search result without the model retyping a URL — added because the model kept fabricating plausible-looking 404 paths), `tab.close`, `tab.list`, `tab.wait_loaded`, `tab.screenshot`, `aria.extract`, `tab.read_active`, `search`, `vision.read` (screenshot + multimodal fallback when the a11y tree is too thin), `tab.click`, `tab.type`, `tab.select`, `tab.scroll`, `tab.upload_file`.

- **ARIA simplification** (`tools/browser/aria.ts`): recursively drops layout-only nodes, splices unnamed wrapper nodes up a level, collapses single-child wrapper chains, assigns a sequential integer index to every interactive node, and caps serialized text at 12,000 chars via escalating trims that specifically protect form inputs even inside otherwise-collapsed nav/footer landmarks. The index→node cache is keyed by tab+URL; invalidated on navigation and unconditionally after every successful mutating action — "act, then re-read" is a structural assumption, not a live DOM diff.
- **`tab.scroll`** has no domain-tier gate (it's read-only) but clears the extraction cache unconditionally after every scroll and reports honest before/after `scrollY` deltas — infinite-scroll/lazy-load pages can mutate the DOM, so the cache is invalidated defensively rather than diffed. This is deliberate, not a bug (a stale code comment once asserted scrolling "changes nothing"; a 2026-07-01 review corrected the comment but left the behavior as-is since no live failure had reproduced a problem).
- **Upload** (`tools/browser/upload.ts`): Chrome blocks `DOM.setFileInputFiles` for extensions, so stored résumé bytes are rebuilt into a `File` client-side and injected via a `DataTransfer` assigned to `input.files`, dispatching `input`+`change` with `composed:true`. The input-finder walks light DOM and **open** shadow roots (closed roots are unreachable to any DOM API, not a bug); cross-origin iframes aren't traversed at all. Never submits the form.
- **Search** (`tools/browser/search.ts`): no API key. Scrapes DuckDuckGo's html/lite endpoints, detects being blocked (captcha/anomaly page) distinctly from a genuine empty result, and as a last resort opens an invisible backgrounded Google tab (full cookie/JS session, so it isn't served a captcha) and scrapes via a per-engine selector cascade.
- Every action tool checks `isElementConnected` before acting and returns an explicit "stale, re-extract" message rather than silently no-op'ing. `tab.select` reads its value back after assignment since an invalid option is a silent no-op otherwise. Clicks use native `.click()`, not synthetic mouse events, because synthetic coordinates didn't reliably navigate on background tabs.

## Safety layer

- **Domain tiers** (`agent/safety/domain_tiers.ts`): two tiers, `read-only` (default, ordinal 0) and `click-only` (ordinal 1). Unknown/unparseable hosts fail closed to `read-only`. `assertCanAct(url, required, tiers, bypass)` checks in this exact order: (1) `isBlockedUrl` — throws unconditionally, before bypass is even read; (2) `if (bypass) return` — skips the tier comparison only; (3) tier comparison. `BLOCKED_PROTOCOLS` = `file:, chrome:, chrome-extension:, devtools:, view-source:, javascript:, data:, blob:, about:, ws:, wss:` — no setting can relax this set. A `full-action` third tier existed pre-2026-07-01, was confirmed byte-identical to `click-only` in enforcement, and was removed; `migrateLegacyTier` reads old persisted `full-action` values back as `click-only` so existing installs aren't silently narrowed. **The same blocklist-then-bypass ordering is composed at a second call site**, `orchestrator.ts`'s private `canActUrl` (gates the consent-auto-dismiss feature only) — it calls the same shared `isBlockedUrl`/`getDomainTier`/`hostFor`/`TIER_ORDER` primitives, so the underlying check can't itself diverge, but the *ordering* is written twice; a future change to one wouldn't automatically apply to the other.
- **Redaction** (`agent/safety/redact.ts`): one shared recursive walker (`redactAny`) backs both `redactEvent` (values only, depth-capped at 8, used on every `TimelineEvent`) and `redactDeep` (keys and values too, uncapped, used on tool findings and the facts ledger). Patterns: Luhn-validated credit card, SSN, email, phone, address — regex/heuristic, so it won't catch PII outside those five shapes (names, DOB, IBANs, etc. are an inherent, undocumented gap). The recursive-walker design specifically fixed a historical bug class: numbers are never stringified before hitting the regex engine, so a `Date.now()` timestamp can no longer false-positive as a credit-card candidate (it used to, ~10% of the time, under the old per-field checklist approach). Redaction happens only at the persistence/telemetry boundary — user profile data is deliberately sent to the LLM **unredacted** for job-apply autofill to work; it's only scrubbed going into IndexedDB/UI events.
- **Circuit breaker** (`agent/safety/circuit_breaker.ts`): pure, stateless-transition functions. Four independent trip conditions checked every turn: 3 consecutive identical actions (exact tool+args hash match — any argument variation evades this), fewer than 3 distinct actions in the last 10 (low diversity), 3+ unknown-tool calls in 8 turns (storm), or 8 turns with no new finding (no progress). `resetForNewStep` discards all windowed counters at step boundaries so a near-miss on one step can't wrongly cut short the next.
- **Prompt-injection defense**: every role's system prompt marks `<untrusted_page_content>` as data, not instructions, applied at one chokepoint (`wrapPageContent` in `prompts/index.ts`).

## Anti-hallucination system

The most deliberate design story in the codebase. `verifyFinish` (`orchestrator.ts`) is **deterministic number-matching**, not another LLM call: normalizes numeric tokens (bare single digits like list markers are exempt) and checks set-membership against a **grounding corpus** = observed-text FIFO (~60KB rolling window) + the facts ledger. An LLM-based "does this hold up against the page" verifier was built and benchmarked, then dropped, because it false-rejected correct/honest answers and dropped the benchmark's `correct` rate from 80% to 67% (`docs/superpowers/specs/2026-06-18-theme-a-page-grounded-verification-design.md`). This is deliberately dumb in a specific way: it's true set-membership on normalized numbers, not substring search, so "4.6" is never wrongly grounded by a page saying "14.62" — but non-numeric hallucinations (wrong names, wrong facts with no digits) are entirely unguarded by this mechanism.

The **facts ledger** (`agent/facts.ts`) holds ≤24 entries, each admitted only if every number inside it is independently grounded at admission time, deduped by exact text, FIFO-evicted. It exists specifically so a fact survives even after its source page scrolls out of the 60KB observed-text window. `mentionsMissing`'s regex (detecting "this field isn't listed" claims) has an unusually long comment trail documenting three rounds of tightening to avoid false-suppression, with one residual false-positive gap left deliberately unfixed (documented, not an oversight).

## Workflow-memory recipes

`agent/workflow_memory.ts` — three origins: `builtin` (16 seed recipes, curated, immutable), `user` (authored via the Recipes UI tab, gets a `lastGood` rollback snapshot), `auto` (learned from a trace, one proven run before disposable, no rollback). `matchWorkflow` scores by keyword/domain overlap with a discriminator gate against false matches. A run is only eligible to *become* a recipe if it was clean (see step 9 above) and its trace isn't redundant (no repeated identical search/open) — a complete user-facing success can still fail to produce or reinforce a recipe if it needed any corrective machinery along the way. Learned-recipe goal text is redacted for PII before persisting (job-apply goals routinely carry email/phone/name).

## Budget & context window

`agent/budget.ts`: `DEFAULT_NUM_CTX` 32,768, range 8,192–131,072, user-configurable in Settings. Per-role token budgets scale with the window, but the **per-page read cap is fixed at 12,000 chars regardless of `num_ctx`** — the stated rationale is that a bigger window should let the agent remember more across turns, not read one page more sprawlingly ("lost in the middle"). Scratch/observed/salvage caps do scale with the window. No runtime guard exists against setting `num_ctx` too high for the user's actual VRAM — that's a documentation-only warning (check `ollama ps` after raising it); misconfiguring it can break every task with no in-app validation. `metrics.ts` only times the 4 LLM role calls, not individual CDP/tool actions, which limits its use for diagnosing whether slowness is the model or the browser layer.

## Job-apply flow

Résumé upload → text extraction (`sidepanel/resume.ts`: mammoth for `.docx`, pdfjs for `.pdf`, 20MB cap; scanned/image PDFs yield empty text with no OCR fallback, surfaced to the user as an explicit "needs OCR" message, no remediation path) → one-shot LLM call extracts profile JSON (`agent/profile.ts`, validation is hand-rolled type/array/empty-value filtering — no Zod, despite Zod being used elsewhere) → user reviews and must explicitly click Save (not auto-persisted). Paste a Greenhouse/Lever URL → `sidepanel/apply.ts` builds a goal string engineered to keyword-match the `seed-job-application` recipe and to textually forbid submission → the agent fills the form and attaches the résumé (same `DataTransfer` injection as general upload) → **stops before submit**.

**That stop is enforced only by prompt/recipe wording — not structurally.** No `submit` tool exists, but generic `tab.click`/`tab.type(..., submit: true)` do, and nothing prevents clicking a button labeled Submit; `ToolContext` doesn't carry a goal or task-type flag, so no tool can check "this is a job-apply run" before acting. The guarantee is entirely prompt discipline plus the generic domain-tier gate. Not yet supported at all: Workday's drag-drop/direct-to-S3 uploader, and forms embedded in a cross-origin iframe.

## Side panel UI

Single `chrome.runtime.Port` (`sidepanel/port.ts`): `send()` lazily calls `chrome.runtime.connect()` if the port is dead — this both revives a dropped connection and wakes an idle-killed service worker, since `connect()` itself wakes MV3 SWs. Reconnection is on-demand only, never polled — the one exception is a 3.5s Ollama-reachability poll, justified because the extension can't launch `ollama serve` itself. `App.tsx` carries an explicit "contract: do not change message shapes" comment over the connection block. Timeline events are capped at 1,000 to bound memory on long runs. Steering (`agent.steer`) is fire-and-forget from the UI's perspective — no `SwUpdate` confirms it was received.

## Testing, build, bench

- **Unit/integration/property** (`npm test`): vitest + happy-dom + `fake-indexeddb/auto`, `chrome.*` stubbed in-memory in `tests/setup.ts` — no real Chrome needed. Coverage is heaviest on background plumbing (~11 files) and tools/ARIA/upload (~15 files); thinnest on the side panel UI (2 files).
- **Bench** (`npm run bench`, opt-in via `OLLAMA_BENCH=1`): `ScriptedBrowser` fakes CDP via a per-task transition-state-machine (no real Chrome), replays typed field values back into the ARIA string so evaluators can verify a form was actually filled. Runs the **real** `Orchestrator` + **real** local Ollama model against scripted fixtures; `scorer.ts` computes `completed`/`correct`/`grounded` (grounded = passes `ungroundedNumbers` *and* has no leaked-but-unobserved entities/forbidden strings — number-grounding alone can't catch single-digit fabrications like an invented "5 stars").
- `scripts/measure_toolcalls.mjs` and the bench are two different things: the former measures model-only tool-call syntax reliability (no orchestrator); the latter measures full-loop task correctness.
- `scripts/browser_smoke.py` is a separate Selenium sanity sweep against a real, built Chrome extension — light assertions, not a full harness.

HEAD is right after the `prod-readiness-review-fixes` merge, which: collapsed the three-tier domain model to two, centralized redaction into `Orchestrator.emit`, added quarantine handling for `auto`-origin recipes (previously only `user` recipes were quarantined), made mid-run service-worker death visible to the UI (previously silently reset to bare `IDLE`), and fixed upload to pierce open shadow roots.

**Known open gap:** the job-apply submit-guard (see above) — the highest-value fix if asked to harden that flow.

## Recurring design principles

1. **Mechanical beats semantic for this model size.** Every place a ~4B model was asked to make a judgment call it's bad at, the fix replaced the judgment with a rule requiring none: number-grounding instead of LLM-verify; same-domain-anchoring instead of asking the model to classify a comparison's basis; a hardcoded protocol blocklist instead of a prompt rule.
2. **Evidence over a-priori design.** The LLM finish-verifier wasn't dropped speculatively — it was built, benchmarked, measured to regress `correct` 80%→67%, then dropped. Changes to grounding/safety logic in this repo are expected to clear `npm run bench`, not just "sound right."
3. **Fail closed, degrade loud.** Unknown hosts → read-only. Unparseable URLs → blocked. Ungrounded numbers → downgraded with the reason appended inline, never silently asserted. A stale DOM node → an explicit re-read message, never a silent no-op.
4. **Defense in depth, with a real duplication cost.** Domain-tier checks live inside individual tool bodies, not only behind a central gate — so a bug in one layer doesn't remove the other, at the cost of a forgetful future tool silently bypassing the system, and the blocklist-ordering being written at two call sites (see Safety layer above).
5. **Trust calibrated to investment.** Auto-learned recipes (cheap to produce) are deleted on first friction. User-authored recipes (a human's effort) roll back to last-known-good instead. Built-in recipes are untouchable.
