# CSV write/repair — design

## Scoping note (read this first)

"CSV-write-repair" was named as a remaining opportunity in the 2026-07-04 browser-use competitive analysis, but that analysis doc has since been deleted and the name was never discussed or scoped in this project before this spec. There is zero prior CSV-related code, discussion, or git history in this repo (confirmed by searching `git log --all -p -S"csv"` and grepping `docs/`/`src/` — no hits). Everything below is a fresh scope decision, made unprompted per the active instruction to write this spec without pausing for clarification. The reasoning for each call is written out so it can be challenged on review.

**What "CSV-write-repair" turned out to mean, concretely:** browser-use (the reference competitor) has a general multi-format virtual file system (`browser_use/filesystem/file_system.py` — md/txt/json/jsonl/csv/pdf/docx/html/xml, ~700 lines) that an agent can write to. Its `CsvFile` subclass runs every write through Python's `csv` module (parse then re-serialize) specifically because "LLMs frequently produce malformed CSV (missing quotes around fields with commas, inconsistent empty fields, unescaped internal quotes)" — that parse-and-fix pass is the literal feature the name refers to.

Copying that design wholesale would mean building a 9-format general file system this project has never had and doesn't otherwise need — a much bigger feature than "CSV." This spec scopes to **just CSV, just the one real use case a browser agent has for it** (save scraped/extracted tabular data so the user can open it in a spreadsheet), and — see "Why structured rows, not raw text" below — solves the "malformed CSV" problem in a way better suited to this project's tool-calling shape than porting browser-use's parse-and-repair approach verbatim.

## Motivation

The agent can extract information from pages today (`aria.extract`, `vision.read`) but has no way to hand back multi-row tabular data as an artifact — the only outputs are the chat-style `finish` summary (prose, not a spreadsheet) and `memory.write` (a single string value, not tabular, and explicitly "for durable knowledge only," cross-task). A goal like "collect the name and price of every product on this page (and the next 3 pages) into a spreadsheet" has no tool that produces a spreadsheet at the end — today the model would have to paste a table into its prose summary, which this project's own 4B-parameter local model formats inconsistently across even a handful of rows.

## Scope

In scope:
1. A new tool `csv.write_rows` — the agent supplies rows as a structured array (not hand-written CSV text — see below), the tool durably accumulates them per filename for the current task, across multiple calls (`write` to start/replace, `append` to add more — e.g., across paginated results).
2. Deterministic repair: ragged rows (wrong column count) are padded/truncated to the established width rather than corrupting the file or crashing; RFC-4180 quoting/escaping is generated by the tool itself, never by the model.
3. A way for the user to actually get the file: a "Files" affordance on the finished-run result card, backed by a new on-demand background request, producing a real `.csv` download via a Blob + anchor click (no new Chrome permission).
4. Executor prompt guidance: recognize a "collect/scrape/save a list of X" goal and use `csv.write_rows` instead of describing a table in prose.

Out of scope (see Non-goals): other file formats, `chrome.downloads` integration, verifying the *data* in each row is grounded/correct (distinct from this feature — see below), editing/deleting individual rows after the fact, a row/size cap beyond what already exists structurally.

## Why structured rows, not raw CSV text

Two ways the tool's arguments could carry the data:

- **A — `rows: string[][]`** (a JSON array of arrays — Ollama's tool-calling already validates this shape via the JSON Schema before `dispatch()` runs).
- **B — `content: string`**, the model hand-writes actual CSV text (`"name,price\nMouse,$9.99"`), mirroring browser-use's `CsvFile` exactly — the tool then parses and re-serializes it to fix mistakes.

**Decision: A.** This project's own recurring principle, stated repeatedly elsewhere in this codebase (grounding, the anti-hallucination work, the reverted LLM-based verifier), is that **mechanical validation beats trusting a small model's free-text formatting.** Option B's entire premise — "the model will make CSV-syntax mistakes, so parse and fix them after" — is a real problem *for browser-use*, whose action schema is a single opaque `content: string` field reused across all 9 file types (so a raw string is the only option it has). This project's tools are already JSON-Schema-validated structured arguments; asking the model to hand-write comma/quote-escaped text inside one of those arguments would be reintroducing a syntax-error surface this project's tool-calling already eliminates for every other tool. With option A, there is no CSV syntax for the model to get wrong — it emits plain string cell values, and the tool performs 100%-correct RFC-4180 serialization deterministically, every time, in code. "Repair" narrows to the one thing structured args can't prevent: a row with the wrong number of columns (see below) — still handled, just a smaller and different surface than browser-use's.

This was checked empirically, not just reasoned about, per this project's standing practice of validating a tool schema against the real local model before committing to it (same as the form-fill-batching spec). Six trials against real `gemma4:e4b`, given a `csv.write_rows` tool definition and a page with a 4-item product list: **6/6 tool calls made, 6/6 with the correct 4 rows and correct values** (no hallucinated extra rows, no dropped rows, consistent 3-column shape matching the declared headers every time). A follow-up 3-trial check with intentionally incomplete source data (one item missing a price) showed the model consistently kept the row at the correct column count, using an empty string for the missing value, when the tool description explicitly says to do that — confirming the prompt-level instruction is what needs to carry that guidance, not just a hope that the model infers it.

## Ragged-row repair

Even with structured args, a row can still arrive with the wrong number of columns — e.g., the model omits a trailing empty value instead of sending `""`, or a later `append` call in a long multi-page scrape drifts from the header established by the first `write` call. Repair rule, applied per row before it's stored:

- **Fewer cells than the established width:** pad with `''` at the end.
- **More cells than the established width:** truncate extra cells from the end.

Both cases are reported back in the tool's `content` (e.g., `"Wrote 3 rows to products.csv (1 row padded to 4 columns)."`) — visible in the Timeline like any other tool result, never silent. This is the same "fail closed, degrade loud" pattern used elsewhere in this codebase (e.g., the anti-bot pause, the domain-tier error messages) — a shape mismatch never crashes the tool or drops data, but it's never invisible either.

The **established width** comes from `headers` if the first `write` call for a filename provided one, otherwise from that first call's own row length. `headers` is only meaningful on the call that creates the file; a `headers` argument on a later call (`append`, or `write` to an already-existing filename) is ignored — the width and column identity are fixed at creation, matching how a real spreadsheet's columns don't change once data exists.

## Explicitly not in scope: verifying the data itself

This is a syntax/shape guarantee, not a correctness guarantee — deliberately, so it doesn't get confused with this project's actual grounding work (`agent/verify/grounding.ts`, `verifyFinish`/`gateFinishSummary`). Grounding verification exists because a finish summary makes a *claim* the user might act on without checking; a CSV file is closer to a working export the user is expected to open and look at themselves — the same trust level as a screenshot, not the same as a final verdict. Extending mechanical grounding to CSV rows (cross-checking each cell against the facts ledger or observed text) is a legitimate future idea but a meaningfully bigger feature — it would need to handle partial matches, numeric formatting differences, and text the facts ledger's 24-entry cap already ages out — and is called out as a Non-goal rather than attempted here.

## The tool: `agent/tools/csv.ts`

Not under `tools/browser/` — it never touches a tab, CDP, or a page, so it doesn't belong with the click/type/scroll tools. Closer in kind to `memory.write` (`tools/core.ts`): no `tabId`, no domain-tier gating (writing a local export isn't "acting on a domain" — the domain-tier system gates actions taken *against a site*; this tool never sends anything to a site).

```ts
export const csvWriteRowsTool: ToolDefDescriptor<{
  filename: string;
  headers?: string[];
  rows: string[][];
  mode: 'write' | 'append';
}> = {
  name: 'csv.write_rows',
  description:
    "Save extracted data as rows in a CSV file the user can download when the run finishes. Use mode 'write' for the first batch (optionally naming column headers), mode 'append' to add more rows to the same file later (e.g. across paginated results — same file, same column order). If a value is missing on the page, use an empty string for that cell rather than skipping the row or shifting columns.",
  argsSchema: z.object({
    filename: z.string().min(1),
    headers: z.array(z.string()).optional(),
    rows: z.array(z.array(z.string())).min(1),
    mode: z.enum(['write', 'append']),
  }),
  async dispatch({ filename, headers, rows, mode }, ctx) {
    const name = sanitizeCsvFilename(filename);
    const result = await upsertCsvRows(ctx.taskId, name, { headers, rows, mode });
    if (!result.ok) return { ok: false, content: result.error };
    const note = result.repaired > 0 ? ` (${result.repaired} row${result.repaired === 1 ? '' : 's'} padded/truncated to ${result.width} columns)` : '';
    return { ok: true, content: `Wrote ${rows.length} row(s) to ${name} — ${result.totalRows} total row(s) so far.${note}` };
  },
};
```

`sanitizeCsvFilename`/`upsertCsvRows` live in `agent/csv.ts` (pure-ish; `upsertCsvRows` is the one function that touches storage, everything else in that file is a pure function — mirrors `agent/totp.ts`'s split between pure crypto/parsing and the one tool that calls it).

## `agent/csv.ts`

```ts
export interface CsvUpsertResult {
  ok: true;
  totalRows: number;
  width: number;
  repaired: number;
}
export interface CsvUpsertError {
  ok: false;
  error: string;
}

/** Basename only (no path traversal), safe characters only, forces a .csv extension --
 *  this becomes a real suggested filename in the browser's save dialog, so the same
 *  concerns apply as any other user-facing filename. */
export function sanitizeCsvFilename(name: string): string {
  const base = name.split(/[/\\]/).pop() || 'data';
  const withoutExt = base.replace(/\.csv$/i, '');
  const cleaned = withoutExt.replace(/[^a-zA-Z0-9_\-. ]/g, '').trim() || 'data';
  return `${cleaned}.csv`;
}

/** Pads a short row with '' or truncates a long one to `width` cells. Returns the row
 *  unchanged (same reference) when it's already the right width, so callers can cheaply
 *  tell whether a repair happened via reference equality. */
export function normalizeRowWidth(row: string[], width: number): string[] {
  if (row.length === width) return row;
  if (row.length < width) return [...row, ...Array(width - row.length).fill('')];
  return row.slice(0, width);
}

/** RFC 4180 serialization. A cell is quoted only if it needs to be (contains a comma,
 *  double quote, or newline); a quote inside a quoted cell is escaped by doubling it.
 *  Deterministic -- given the same rows, always produces the same, always-valid text. */
export function toCsvText(headers: string[] | undefined, rows: string[][]): string {
  const quoteIfNeeded = (cell: string): string =>
    /[",\n]/.test(cell) ? `"${cell.replace(/"/g, '""')}"` : cell;
  const lines = headers ? [headers, ...rows] : rows;
  return lines.map((line) => line.map(quoteIfNeeded).join(',')).join('\n');
}
```

## Storage: `background/state_store.ts`

A new IndexedDB object store, `csvFiles`, `keyPath: 'taskId'` — one record per task holding every CSV file written during it, exactly mirroring the existing `scratchpad` store's shape (`{ taskId, content, updated }`) and its read-modify-write accessor pair (`getScratchpad`/`setScratchpad`), not the `findings`/`events` append-log shape (autoIncrement id + byTask index) — those are for independent historical records; a CSV file is one mutable object that grows in place across `append` calls, which is exactly what `scratchpad` already models for the (unrelated) turn-notes text.

```ts
export interface CsvFileRecord {
  headers?: string[];
  rows: string[][];
  updated: number;
}

// in the `upgrade()` block, alongside the existing stores:
if (!d.objectStoreNames.contains('csvFiles')) {
  d.createObjectStore('csvFiles', { keyPath: 'taskId' });
}

export async function upsertCsvRows(
  taskId: string,
  filename: string,
  input: { headers?: string[]; rows: string[][]; mode: 'write' | 'append' },
): Promise<CsvUpsertResult | CsvUpsertError> {
  const d = await db();
  const rec = ((await d.get('csvFiles', taskId)) as { taskId: string; files: Record<string, CsvFileRecord> } | undefined) ?? {
    taskId,
    files: {},
  };
  const existing = rec.files[filename];

  if (input.mode === 'append' && !existing) {
    return { ok: false, error: `No existing file named ${filename} — call with mode:'write' first.` };
  }

  const headers = input.mode === 'write' ? (existing ? existing.headers : input.headers) : existing!.headers;
  const width = headers?.length ?? (input.mode === 'write' && !existing ? input.rows[0]?.length ?? 0 : existing!.rows[0]?.length ?? input.rows[0]?.length ?? 0);

  let repaired = 0;
  const normalized = input.rows.map((r) => {
    const n = normalizeRowWidth(r, width);
    if (n !== r) repaired++;
    return n;
  });

  const priorRows = input.mode === 'append' && existing ? existing.rows : [];
  const nextRows = [...priorRows, ...normalized];
  rec.files[filename] = { headers, rows: nextRows, updated: Date.now() };
  await d.put('csvFiles', rec);

  return { ok: true, totalRows: nextRows.length, width, repaired };
}

export async function listCsvFiles(taskId: string): Promise<Array<{ filename: string; rowCount: number }>> {
  const d = await db();
  const rec = (await d.get('csvFiles', taskId)) as { files: Record<string, CsvFileRecord> } | undefined;
  if (!rec) return [];
  return Object.entries(rec.files).map(([filename, f]) => ({ filename, rowCount: f.rows.length }));
}

export async function getCsvFileText(taskId: string, filename: string): Promise<string | null> {
  const d = await db();
  const rec = (await d.get('csvFiles', taskId)) as { files: Record<string, CsvFileRecord> } | undefined;
  const f = rec?.files[filename];
  if (!f) return null;
  return toCsvText(f.headers, f.rows);
}
```

`DB_VERSION` bumps from 2 to 3 for the new object store, matching how earlier stores were added to this same `upgrade()` function (each guarded by its own `if (!d.objectStoreNames.contains(...))`, so existing installs pick up just the new store without disturbing the others).

## Getting the file to the user

No new Chrome permission. `chrome.downloads` would need a manifest change and hands the save dialog to Chrome's download manager; a plain Blob + `<a download>` click — standard web platform behavior, works from any extension page including the side panel — triggers the exact same native save dialog with zero new permissions and zero new manifest surface. Chosen for that reason alone (native platform feature over a heavier API — the same "does this need a new permission" bar this project already applies elsewhere).

**Message protocol** (`shared/messages.ts`), added to the existing `PanelCommand`/response unions, mirroring `session.turnEvents`'s exact shape (a request naming a `taskId`, a response the panel renders):

```ts
// PanelCommand additions:
| { type: 'csv.list'; taskId: string }
| { type: 'csv.download'; taskId: string; filename: string }

// response additions:
| { type: 'csvFiles'; taskId: string; files: Array<{ filename: string; rowCount: number }> }
| { type: 'csvDownload'; taskId: string; filename: string; csvText: string }
```

**Background** (`background/index.ts`), mirroring `handleSessionTurnEvents`'s one-line-body style:

```ts
async function handleCsvList(taskId: string) {
  broadcast({ type: 'csvFiles', taskId, files: await listCsvFiles(taskId) });
}
async function handleCsvDownload(taskId: string, filename: string) {
  const csvText = await getCsvFileText(taskId, filename);
  if (csvText !== null) broadcast({ type: 'csvDownload', taskId, filename, csvText });
}
```

**Side panel:** `ResultCard` (the finished-run summary — already shows verdict/summary/sources/steps, the natural place for "here's what this run produced") gets two new optional props, `files` and `onDownload`, rendering a small block mirroring its existing `result-sources` block:

```tsx
{files.length > 0 && (
  <div className="result-files">
    <Icon name="download" size={12} />
    {files.map((f) => (
      <button key={f.filename} className="btn btn-ghost btn-sm" onClick={() => onDownload(f.filename)}>
        {f.filename} ({f.rowCount} rows)
      </button>
    ))}
  </div>
)}
```

`App.tsx` requests `csv.list` whenever a run finishes (same trigger point that already exists for other finish-time bookkeeping), stores the response in new `csvFiles` state, passes it to `ResultCard`, and on `csvDownload` performs the actual browser download:

```ts
case 'csvFiles':
  setCsvFiles(msg.files);
  break;
case 'csvDownload': {
  const blob = new Blob([msg.csvText], { type: 'text/csv' });
  const url = URL.createObjectURL(blob);
  const a = document.createElement('a');
  a.href = url;
  a.download = msg.filename;
  a.click();
  URL.revokeObjectURL(url);
  break;
}
```

Revisiting a past session (via `session.select` → `session.turnEvents`, the existing mechanism) also re-requests `csv.list` for that taskId, so a CSV written in an earlier run stays downloadable after switching sessions — not just immediately after the run that created it.

## Testing

`tests/unit/csv.test.ts` (pure functions, no chrome/IDB mocking needed):
- `toCsvText`: a plain field is unquoted; a field containing a comma is quoted; a field containing a double quote is quoted with the quote doubled; a field containing a newline is quoted; an empty `rows` array with headers produces just the header line; matches a hand-checked multi-row example exactly.
- `normalizeRowWidth`: a short row is padded with `''`; a long row is truncated; a row already the right width is returned unchanged (same reference, verified with `toBe`, not `toEqual`).
- `sanitizeCsvFilename`: adds `.csv` when missing; strips a path traversal attempt (`../../etc/passwd` → a bare safe basename ending in `.csv`); strips disallowed characters; a name that's already a clean `.csv` filename round-trips unchanged.

`tests/unit/csv_store.test.ts` (exercises the real IDB, matching how `state_store.test.ts` already exercises `scratchpad`/`findings` — this project's tests stub IndexedDB in-memory, not chrome-only):
- `upsertCsvRows` with `mode:'write'` and headers, then `listCsvFiles` shows one file with the right row count.
- A second `upsertCsvRows` call with `mode:'append'` on the same filename accumulates rows (total count grows; earlier rows preserved).
- `mode:'append'` on a filename that was never `write`-created returns `{ok:false}` with a clear error, not a throw.
- A `write` call whose first row has 3 cells, followed by an `append` call whose row has only 2, results in a stored row padded to 3 (via `repaired: 1` in the write's own return, and the padded shape visible via `getCsvFileText`).
- `getCsvFileText` on an unknown filename returns `null`; on a known one returns text matching `toCsvText`'s own output for that data (i.e., the store and the serializer agree — not re-asserting `toCsvText`'s formatting rules a second time).
- Two different filenames under the same `taskId` stay independent (writing to one doesn't affect the other's row count).

`tests/unit/csv_tool.test.ts` (mirrors `actions_readback.test.ts`'s tool-test style):
- `csvWriteRowsTool.dispatch` with `mode:'write'` returns `ok:true` with a content message naming the row count.
- A second call with `mode:'append'` reports the cumulative total in its content message.
- `mode:'append'` with no prior `write` for that filename returns `ok:false` with the store's error message surfaced verbatim.
- A ragged row triggers the "padded/truncated" note in the returned `content`.
- Schema rejects a `rows` array with zero elements (`.min(1)`) and a missing `mode`.

## Non-goals

- **Other file formats** (md/txt/json/pdf/docx/html/xml — browser-use's other 8 types). Nothing about this project's current tool set needs them; CSV is the one format tied to a concrete, common browsing-agent task (tabular scraping). Out of scope structurally, not by oversight.
- **`chrome.downloads` API / a new manifest permission.** The Blob+anchor approach covers the same user-facing outcome (a real file lands via the browser's save dialog) with zero new permission surface.
- **Verifying the correctness of row *data*** (grounding each cell against facts/observed text). See "Explicitly not in scope: verifying the data itself" above — this is a syntax/shape guarantee, deliberately not conflated with this project's actual anti-hallucination grounding work.
- **Editing or deleting individual rows/files after they're written.** The model can only write/append; a user who wants to fix the data edits the downloaded file in a spreadsheet like any other export. If a rewrite is needed mid-run, `mode:'write'` again for the same filename already replaces it outright.
- **A row-count or byte-size cap specific to this tool.** The existing circuit breaker (`orchestrator.ts`'s `recordAction`/`breaker`, already generic across every tool) already bounds runaway repeated tool calls, and a single call's row count is naturally bounded by how much the model can extract from its own ~60KB observed-text window in one turn. Adding a second, CSV-specific cap on top would be guarding against a failure mode nothing here suggests is real.
- **Concurrent-write races on the same filename.** The orchestrator's loop calls exactly one tool per turn (documented in `CLAUDE.md`'s "goal's journey" — "every turn calls the helper seat for one tool call"), so two `csv.write_rows` calls for the same task can't execute concurrently; no locking is needed beyond IndexedDB's own per-request atomicity.
